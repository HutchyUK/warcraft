## Problem Statement

Managing multiple WoW Classic Anniversary characters ("alts") is a spreadsheet problem
in disguise. Weekly resets bring a batch of tasks — raid lockouts, dungeon runs,
profession cooldowns, quest turn-ins, rep items — and with 4+ alts you're running a
mental model of "did I do X on Y character this week?" that fails the moment you log
off for a few days.

No tool exists that serves this specific need for Classic Anniversary. Existing tools
(Raider.IO, WoWProgress, ClassicArmory) are public-facing leaderboards, not personal
dashboards. The dominant "solution" is still a spreadsheet or a Blizzard forum thread.

The user wants something they can check on a phone during their lunch break and
immediately know what's left to do before reset.

**Scope:** TBC Classic Anniversary only for v1. Classic Era (vanilla) characters are
out of scope — task lists are too different to model together without complicating
the data model prematurely.

## What Makes This Cool

**The War Room framing:** A single dashboard styled like a WoW raid bulletin board.
Each alt is a class-colored card. Completed raid lockouts get a "CLEARED" visual stamp.
Profession cooldowns show countdowns to their next reset. Gear slots highlight when a
BiS upgrade is available from content you haven't done yet.

**Cross-alt dependency tracking (the novel feature):** Classic players run alts as a
machine serving a main. Your Mage farms Arcanite Transmutes for your Warrior's BiS
gear. The dashboard surfaces these relationships — when the Mage has an unchecked
Transmute task this week, the Warrior card shows: "Mage has a pending task for you."
No spreadsheet and no existing tool does this.

**Anxiety reduction over information density:** Every design decision optimized for the
"30-second glance" — you open it, you see red (needs doing) and green (done), you
close it. Not a wiki, not a database. A status board.

## Constraints

- TBC Classic Anniversary only (v1 scope — not Classic Era, not Retail, not Hardcore)
- Blizzard Classic API is unreliable for most useful data — only character metadata
  (name, class, level, realm, equipped gear snapshot) is trustworthy
- Weekly task data (lockouts, quest completion, profession CDs) must be manual entry
- Weekly reset cadence: Tuesday midnight UTC (US) / Wednesday midnight UTC (EU).
  Characters carry a `region` field; reset boundary is derived from it.
- Some profession CDs (Arcanite Transmute) are 2-day cooldowns, not weekly — these
  need their own reset logic independent of the weekly Tuesday/Wednesday boundary
- Side project — no budget for expensive infra, should be free/near-free to run
- Turso free tier: ~500M row reads/month (not per-day) — generous for personal use;
  verify current limits at turso.tech before launch

## Premises

1. The Blizzard API is used only for character metadata (name/class/level/realm/gear).
   Manual entry handles weekly tasks. Classic API does not reliably expose lockout
   status, quest completion, or profession cooldowns.
2. Web app, not native/mobile — no install required, works on any device.
3. Weekly reset cadence is the core rhythm for raid lockouts; profession CDs have
   their own period (2-day, 3-day, or weekly) tracked separately.

## Approaches Considered

### Approach A: Local-First SPA
Single-page React app, localStorage only, no backend. Blizzard OAuth for character
import only. No server, deploys to GitHub Pages.
- Effort: S | Risk: Low
- Con: No cross-device sync, data lost on browser clear

### Approach B: Full-Stack Web App
Next.js + Turso SQLite + Drizzle + Tailwind + NextAuth with Blizzard OAuth.
Characters in DB, checklist state persists per user, deploy to Vercel.
- Effort: M | Risk: Low-Med

### Approach C: The War Room (full vision) — CHOSEN
Stack: Next.js 14 (App Router) + Turso (SQLite) + Drizzle ORM + Tailwind CSS +
NextAuth.js with Blizzard OAuth provider. Deploy on Vercel (Node.js runtime, not
Edge — required for Drizzle's libsql adapter compatibility).
Plus: cross-alt dependency tracking, class-themed visual design, BiS gear tracking
with dungeon source mapping, quest chain progression modeling.
- Effort: L | Risk: Med

## Recommended Approach

**Approach C — The War Room.**

Stack: Next.js 14 (App Router) + Turso + Drizzle ORM + Tailwind CSS + NextAuth.js
with Blizzard OAuth. Deploy to Vercel using the **Node.js runtime** for all API routes
(not Edge runtime — Drizzle's `@libsql/client` has known edge-runtime incompatibilities
on Vercel). Add `export const runtime = 'nodejs'` to each route handler.

### Core data model

```
User
  id, blizzard_account_id, battletag, created_at, updated_at

Characters
  id, user_id, name, realm, class, level, role, is_main,
  blizzard_char_id, region (US|EU), avatar_url, spec

WeeklyTasks
  id, character_id, task_key, task_name, task_type (RAID|WEEKLY_QUEST),
  week_start (DATE — the Tuesday/Wednesday that began this reset period),
  checked_at (TIMESTAMP, nullable — NULL = unchecked, non-NULL = checked)
  [Heroic daily lockouts are NOT stored here — they reset daily, not weekly]

DailyTasks
  id, character_id, task_key, task_name, task_type (HEROIC_DAILY),
  day_start (DATE — the UTC calendar day this entry covers),
  checked_at (TIMESTAMP, nullable)
  [One row per heroic per character per day: heroic_shadow_labs, heroic_steam_vault, etc.]

ProfessionCooldowns
  id, character_id, cd_key, cd_name, period_days (2 or 3 or 7),
  last_used_at (TIMESTAMP, nullable — NULL = never used)
  [reset = last_used_at + period_days, independent of weekly cadence]

GearSlots
  id, character_id, slot_name, current_item (text), bis_item (text),
  bis_source (text — e.g. "Karazhan - Moroes"), is_complete (BOOLEAN)
  [is_complete = true when current_item matches bis_item or player manually marks it done]

QuestProgress  [Phase 2 only — do not create this table in Phase 1]
  id, character_id, chain_key, step_key, step_name, completed_at

AltDependencies
  id, source_character_id, target_character_id, description (text)
  [descriptive only — "Mage → Warrior: Arcanite Transmute"]
  [badge fires when source character has any WeeklyTasks or ProfessionCooldowns
   that are unchecked/off-cooldown this period]
```

**Weekly reset logic for `WeeklyTasks`:**
- Compute `current_week_start`: last occurrence of Tuesday midnight UTC (US chars) or
  Wednesday midnight UTC (EU chars) based on `Characters.region`
- Store a hardcoded task template per TBC phase (a static list of task_keys for raids
  and weekly quests). On dashboard load, compute the task list as:
  `template_tasks LEFT JOIN WeeklyTasks WHERE week_start = current_week_start`
  — tasks not yet in the DB for this period render as unchecked. First check action
  inserts the row. This avoids a blank dashboard on the first visit of each new week.
- Dashboard query joins Characters + this week's WeeklyTasks + DailyTasks (today) +
  ProfessionCooldowns (current) in a single query per user.

**Daily reset logic for `DailyTasks`:**
- `day_start = current UTC calendar date`
- Dashboard query: `WHERE day_start = today` — heroics not in DB today render as unchecked

**Profession CD reset logic:**
- `is_on_cooldown(cd) = last_used_at + period_days > now`
- "Use now" button sets `last_used_at = now`
- No week_start dependency

### Phase 1 — Foundation (Weekend 1)

**Friday night:**
- Set up env vars: `BLIZZARD_CLIENT_ID`, `BLIZZARD_CLIENT_SECRET`, `TURSO_DATABASE_URL`,
  `TURSO_AUTH_TOKEN`, `NEXTAUTH_SECRET`, `NEXTAUTH_URL`. Configure in Vercel dashboard
  and in `.env.local` for dev.
- Wire NextAuth with Blizzard OAuth provider. Access token from OAuth session can be
  used for subsequent API calls until it expires (Blizzard tokens expire in 24h;
  re-authenticate via NextAuth session refresh).
- Fetch Classic character list:
  `GET https://us.api.blizzard.com/profile/user/wow?namespace=profile-classic1x-us&locale=en_US`
  for US; swap `us` for `eu` and `classic1x-us` for `classic1x-eu` for EU.
  Filter response: only store characters where `game_version.type == "CLASSIC"`.
  (Note: field name — confirm against live API response; may vary by Classic version.
  Fallback: if the namespace returns an error or zero Classic characters, fall back to
  a manual character-add form where the user types name/realm/class/level directly.
  The API import is a convenience, not a blocker — the tool works fine with manual entry.)
- Drizzle schema: `User`, `Characters` (with `region`), `WeeklyTasks`, `ProfessionCooldowns`,
  `GearSlots` tables. Defer `QuestProgress` and `AltDependencies` to Phase 2.
- Character cards on dashboard, color-coded by class

**Saturday:**
- Hardcode TBC Classic Anniversary task list (current phase at build time):
  - Raids: Karazhan (1 lockout per character), Gruul's Lair, Magtheridon, SSC, TK
  - Heroic Daily: each heroic dungeon has its own daily lockout — stored in `DailyTasks`
    (not WeeklyTasks), with `day_start = today`. Individual task keys per dungeon:
    `heroic_shadow_labs`, `heroic_steam_vault`, `heroic_mana_tombs`, etc.
  - Weekly quests (hardcode for current phase; update the list as phases advance)
- Weekly checklist with region-aware reset, lazy row insertion, `checked_at` semantics
- ProfessionCooldowns with per-CD period logic (Arcanite Transmute = 2 days)
- Mobile-readable layout: green checkmarks, red circles

**Sunday:**
- GearSlots tracker: free-form text (slot + current + BiS + source), `is_complete` toggle
  meaning "BiS acquired or no longer needed"
- "Needs" rollup card: all characters with `is_complete = false` gear rows
- Deploy to Vercel, set env vars in dashboard, provision Turso DB
- Verify on phone

### Phase 2 — The Differentiating Features (Weekend 2-3)

- **Cross-alt dependency tracking**: link characters with a description; badge on
  source character when they have unchecked tasks; badge on target character
  showing who is farming for them
- **Quest chain progression**: `QuestProgress` table; hardcode major TBC attunement
  chains (SSC/TK, Black Temple, Hyjal). Mark steps complete per character.
- **Visual polish**: CLEARED stamps, class color theming throughout, phase-aware task
  list with a settings toggle for "current phase"

### Phase 3 — Quality of Life (ongoing)

- Phase-aware task management: add a "current phase" setting; task list filters to
  only show content available in that phase (Karazhan phase vs. Black Temple phase)
- Task template import/export as JSON — share your checklist config
- PWA manifest (`manifest.json` + meta tags, ~20 min) for phone homescreen install

## Success Criteria

- On any device, within 30 seconds, the user can see which alts have outstanding
  weekly tasks and which are fully cleared
- After raid night, checking off lockouts takes less than 60 seconds
- At least one instance where the cross-alt dependency view surfaces something the user
  would have forgotten
- The tool is actually opened more often than a spreadsheet would be

## Reviewer Concerns

The following issues were flagged across 3 adversarial review iterations. The 3 high-severity
items were fixed in the document above. These medium/low items are acknowledged and deferred:

1. **BiS cross-reference not scheduled** (Medium): "What Makes This Cool" promises that gear
   slots highlight when an uncompleted task drops your BiS item. The implementation phases
   never schedule this cross-reference between `GearSlots.bis_source` and task completion.
   This is the most visually interesting feature and needs a dedicated Phase 2 implementation
   step. Suggested: add a computed `needs_from_lockout` flag derived by matching `bis_source`
   text against known raid/dungeon names from the task template.

2. **NextAuth refreshAccessToken callback** (Medium): Blizzard OAuth tokens expire in 24h.
   If a user stays logged in across a day, API calls for character refresh will fail.
   The NextAuth Blizzard provider needs a `refreshAccessToken` callback. This is a one-time
   implementation detail but easy to forget.

3. **Phase 2 vs Phase 3 phase-awareness overlap** (Medium): "Visual polish: phase-aware task
   list with settings toggle" is listed in Phase 2; "Phase-aware task management" is also
   in Phase 3. Clarify at implementation time: Phase 2 = toggle UI + filtering; Phase 3 =
   automatic phase detection or notification.

4. **AltDependency badge logic location** (Low): The badge trigger condition is only in a
   schema comment. At Phase 2 implementation, write the logic explicitly: badge fires when
   `source_character` has any row in `WeeklyTasks` or `ProfessionCooldowns` for the current
   period where `checked_at IS NULL`.
