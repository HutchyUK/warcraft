# ADR-0003: Authentication — Blizzard OAuth + Cookie Session

**Status:** Accepted  
**Date:** 2026-04-04

---

## Context

The app needs to identify users and gate all data access to the authenticated user's
characters. The natural identity provider is Blizzard Battle.net — users already have
accounts, and OAuth login also grants the `wow.profile` scope needed for character import.

Two token-handling approaches were considered:

| Approach | Pros | Cons |
|----------|------|------|
| JWT (stateless) | No server-side session storage; easy to verify on any server | Token revocation is hard; `blizzard_access_token` would travel in every request header; 30-day expiry requires refresh token flow |
| Cookie session (stateful) | HttpOnly cookie — JS can't read it (XSS protection); sliding expiry is simple; access token stored server-side in claims | Requires `SameSite` config for cross-origin; doesn't scale horizontally without sticky sessions or distributed cache |

At this project's scale (single user, single server instance), the stateful cookie
approach is clearly correct.

## Decision

Use **ASP.NET Core Cookie Authentication** with **Blizzard OAuth** as the external
provider. The Blizzard access token is stored as a claim in the encrypted cookie — it
is never sent to the frontend.

Key configuration choices:

- `SameSite = None` + `Secure = Always` — required for cross-origin requests from
  Vercel (frontend) to Railway (backend). Browsers enforce that `SameSite=None` cookies
  must also be `Secure`.
- `HttpOnly = true` — prevents JavaScript from reading the session cookie.
- `ExpireTimeSpan = 30 days` with `SlidingExpiration = true` — users stay logged in
  without re-authenticating if they use the app regularly.
- CORS `AllowCredentials()` + explicit origin list — required for the browser to send
  the cookie on cross-origin `fetch` calls with `credentials: 'include'`.

### Post-OAuth redirect

After the Blizzard OAuth callback, the backend redirects to `FrontendUrl` (config key).
This must match the Next.js origin exactly. Local dev: `http://localhost:3000`. Production:
the Vercel deployment URL.

### Local development HTTPS requirement

`SameSite=None` requires `Secure` cookies. `Secure` cookies are only sent over HTTPS.
Therefore local development must use the HTTPS profile (`https://localhost:7211`):

```bash
dotnet dev-certs https --trust   # once per machine
dotnet run --launch-profile https
```

The Blizzard redirect URI registered in the developer portal must also use HTTPS
(`https://localhost:7211/api/auth/callback`).

## Consequences

**Positive:**
- No password management — all identity delegated to Blizzard
- `blizzard_access_token` claim is available server-side for character import without
  any additional token storage table
- XSS cannot steal the session (HttpOnly)
- Simple implementation — ASP.NET Core's built-in OAuth middleware handles the full flow

**Negative:**
- Local dev requires trusting a dev HTTPS certificate — one extra setup step
- Blizzard access tokens expire in 24 hours. If a user keeps a session open across a
  day, the stored `blizzard_access_token` claim becomes stale. Character import will
  fail silently until the user re-logs in. A `refreshAccessToken` flow was not
  implemented in v1 (see Reviewer Concerns in DESIGN.md).
- `SameSite=None` cookies are blocked by some aggressive browser privacy settings
  (Brave's shields, Firefox Strict mode). Users who see login failures should check
  their browser settings.

**Neutral:**
- The `warcraft_session` cookie name is arbitrary. It is set in `Program.cs` and has
  no functional significance beyond being recognisable in browser devtools.
